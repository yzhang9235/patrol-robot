# 一. IDC 巡检摄像头

## 安装依赖
```bash
pip install opencv-python numpy
```
--------------------------------------------------------------------------------
## 使用步骤
# 1. 标定LED位置
```bash
python calibrate.py
```
- 鼠标拖拽框选每个"灯"，输入名字（比如 `server_01_power`, `server_02_status`）
- 按 `s` 保存 -> 生成 `regions.json`

# 2. 开始监控
```bash
python monitor.py
```
- 实时画面上会标出每个区域的当前颜色和是否闪烁
- 触发告警时终端会红色高亮打印，并写入 `alerts.log`
--------------------------------------------------------------------------------
# 告警逻辑（需要根据实际的闪频规则修改）
| 灯的状态  | 告警内容 |
| 红灯常亮  | Power Supply Failure（疑似电源故障） |
| 红灯闪烁  | 严重故障，需要人工确认 |
| 灯灭     |  可能断电或传感器异常 |
| 黄灯闪烁  | 警告状态，建议关注 |
| 绿灯常亮  | 正常，不告警 |

--------------------------------------------------------------------------------
# 通向实际摄像头的方法
- 把 `cv2.VideoCapture(0)` 换成 RTSP流地址（真实IP摄像头），比如 `cv2.VideoCapture("rtsp://user:pass@ip:554/stream1")`
- 告警输出这里可以写入运营管理平台的告警接口
- Phase2 要做的"server型号识别 + LED pattern database"，可以在 `evaluate_alerts()` 
这个函数里替换成查表逻辑：先识别server型号 -> 查该型号的LED含义表 -> 再判断当前颜色/闪烁模式
对应哪种故障

# 二. spider机器人
###### ArUco地标方法 #######
## 使用步骤
## 1.生成标记
```bash
python generate_markers.py
```
- 先把代码顶部 RACK_LIST 改成实际的机柜编号（例的A01-A04），跑完会生成：marker_0.png ~ marker_3.png 的二维码
- 打印出来，贴在对应机柜上（建议打印≥10cm×10cm，太小的话摄像头稍微远一点就识别不出来）
rack_positions.json —— marker编号和rack_id的对照表

## 3.模拟机器人找目标
```bash
python locate_marker.py A03
```
- 把打印好的marker贴在模拟机柜，拿对应摄像头对着找，画面里会实时显示：
- 找到目标marker时：终端打印"方向正确继续前进" / "目标偏左/偏右X px，向左/右转" / "已到达，
停止移动"
（没指定rack_id直接跑 python locate_marker.py，会显示画面里所有识别到的marker和对应rack_id，方便先测试识别效果）

## 怎么接到真实机器人
现在脚本只是在终端打印方向指令（"向左转"/"向右转"/"已到达"），这部分逻辑本身就是机器人要用的
导航算法核心。真正接入硬件时，需要把 guidance_for_marker() 返回的这几种状态，换成实际
发给电机的指令，比如：
if "向右转" in msg:
    robot.turn_right(speed=30)
elif "向左转" in msg:
    robot.turn_left(speed=30)
elif "继续前进" in msg:
    robot.move_forward(speed=50)
elif "已到达" in msg:
    robot.stop()
    robot.take_photo()  # Phase 3要的拍照+OCR就从这里接进去
这样"用摄像头看marker决定往哪走"这套视觉逻辑跟"具体怎么驱动轮子/腿"完全解耦了——不管spider 
robot用的是什么电机驱动板，接进去都不用改上面这段识别逻辑。
